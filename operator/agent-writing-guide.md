# Joel-Facing Writing Guide

Use this guide when writing for Joel or explaining York ExAI from Joel's point of view.

The goal is not to teach him software.

The goal is to make plain what the system handles for him, what it protects the business from, and what is real today versus still being built.

## Core Rule

Write like a dependable coworker explaining how the business side should get easier.

Do not write like a product team, a software trainer, or a helpdesk.

## What Joel-Facing Docs Are For

Joel-facing docs should:

- explain what the system is supposed to take off Joel's plate
- describe daily, weekly, and monthly relief in business terms
- tell the truth about what already works and what still needs built
- build trust without overselling unfinished behavior

Joel-facing docs should not:

- turn into CLI instructions
- explain implementation details unless Joel actually needs them
- sound like onboarding copy for a SaaS product
- re-teach pest control

## Present-State And Future-State Rule

Keep three labels clear:

- `working now`: behavior already backed by current implementation and validation
- `partially there / supervised`: behavior that exists but still needs review or close handling
- `coming next`: intended near-term direction that is not yet something Joel should rely on as finished

Do not blur these together.

If a behavior is planned, say it is planned.

If a behavior is real but supervised, say that too.

## What To Talk About

Lead with business reality:

- route pressure
- callbacks
- paperwork gaps
- voice memos and photos
- missed details
- clean versus risky closeout
- invoice readiness
- loose ends at end of day, week, or month

Treat the operational contract plainly:

- Joel does the pest-control work and gives quick field signal
- the system captures, organizes, flags, and hands work forward
- incomplete compliance-sensitive work must not look finished
- callback work must not quietly become clean revenue

## What Not To Talk About

Avoid these unless an internal audience truly needs them:

- command syntax
- table names
- schema design
- event contracts
- JSON envelopes
- storage layout
- implementation stack details

Those belong in technical docs, not Joel-facing explanation.

## Tone Rules

Follow `context/voice-and-tone.md`, `context/human-roles.md`, and `domain/customer-communication.md`.

Practical rules:

- short, plain, direct
- coworker tone
- low fluff
- no fake blue-collar performance
- no motivational filler
- no "software transformation" language
- respect Joel's trade judgment

## Banned Phrasing

Do not use language like:

- workflow optimization
- dashboard visibility
- operational synergy
- seamless user experience
- end-user adoption
- frictionless platform
- digital transformation
- just click here
- the system empowers you

## Better Phrasing

Prefer language like:

- this should keep the loose ends from piling up
- this should help keep the route straight
- this should hold onto the job detail so it does not get lost
- this should stop sloppy paperwork from looking finished
- this should help the office get the billing side handled without dragging Joel back into it

## Good And Bad Examples

Good:

- "You should not have to babysit this thing all day."
- "If the paperwork is not there, it ought to stay blocked."
- "If you send a voice memo, the system should keep it and sort it out without losing it."

Bad:

- "This workflow gives the user visibility into outstanding compliance tasks."
- "The platform streamlines route execution through a unified command surface."
- "Joel can leverage the interface to optimize documentation throughput."

## Content Boundary

The main Joel-facing doc is expectation-oriented, not instruction-oriented.

That means:

- explain what York ExAI should do for Joel
- explain what kind of signal it needs from him
- explain what is already dependable and what is still coming

Do not make Joel feel like he is responsible for operating the admin machinery.
